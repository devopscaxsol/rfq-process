üìÑRFQ Handling Process (Technical Team View)

A structured and repeatable process to manage client RFQs within the company:

üß≠ Phase 1: Intake and Internal Review

Step Description
1. RFQ Registration Sales, BD, or Procurement team logs the RFQ into a centralized tracking system and assigns a unique RFQ ID.
2. Internal Kick-off Meeting Meeting involving Sales/BD, Technical Architect, Project Manager, and stakeholders to:
  • Understand scope
  • Identify deliverables
  • Define timeline
  • Assign team leads
3. Scope Classification Technical Architect categorizes the RFQ by:
  • Type: Software / Infrastructure / Integration / Support
  • Complexity: Small, Medium, Complex
  • Domain or Vertical

üîç Phase 2: Technical Analysis

Step Description
4. Requirements Breakdown Technical team extracts and validates:
  • Functional & non-functional requirements
  • Compliance and integration needs
  • Expected deliverables (documents, code, designs)
5. Gap Analysis & Risk Assessment Identify:
  • Requirement gaps
  • Risks related to technology, timeline, dependencies
  • Ambiguities needing clarification
6. Clarifications to Client Technical team sends queries to BD or directly to client to resolve ambiguities.

üí° Phase 3: Solutioning and Estimation

Step Description
7. Solution Proposal Architect prepares a high-level solution including:
  • Architecture diagram
  • Proposed tech stack
  • Integration strategy
8. Effort Estimation Leads from Development, Infrastructure, Security, QA provide:
  • Effort estimation (hours or person-days)
  • Resource planning
9. Costing Combine effort estimates with financial rate cards (via PM/Finance) to determine pricing.
10. SLA/Timeline Commitments Define:
  • Project timeline with milestones
  • Service Level Agreements (SLAs) or performance targets
  • Client-side dependencies

üìë Phase 4: Documentation and Review

Step Description
11. Prepare Technical Sections Draft technical proposal sections including:
  • Technical approach
  • Compliance checklists
  • Assumptions and exclusions
12. Internal Review & Approval Review and approval by Technical Architect, Delivery Head/CTO, and Commercial Owner.
13. Final Submission Merge technical and commercial proposals and submit via BD/Procurement team.

‚úÖ Phase 5: Post-Submission Activities

Step Description
14. Response to Clarifications Technical Points of Contact support follow-up clarifications, demos, or presentations.
15. Handover to Delivery If RFQ is awarded, hand over technical documentation and commitments to Delivery/PMO.

Role Responsibilities
Technical Architect Lead technical review, define architecture, own effort estimation.

Sales / BD Coordinate RFQ intake and client communications.
PMO / Delivery Manager Consolidate proposals and resource planning.
